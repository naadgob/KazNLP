Путь исследования

Какие подходы пробовали для шала-казахского, что показали метрики и почему пришли к XLM-R + маршрутизации тона.

Главный урок

Правила по буквам и лёгкие классификаторы дают тысячи ложных mixed. Нужен контекст (трансформер) + ручной gold.

01

Эвристика по буквам

Отклонён
Подход

«Есть ә/ң/ғ + кириллица → mixed»; чистый kz без казахских букв → не kz.

Результат

Ловит только узкий слой; пропускает шала-казахский на русской раскладке.

Вывод

Правила по орфографии не масштабируются на соцсети — отказались от LID на правилах.

02

FastText v1 (синтетика)

Baseline
Подход

~0k синтетических строк (ru/kz/mixed), словари ru↔kz, обучение fasttext_v1.bin.

Результат

~0% F1 на synthetic test; в Telegram десятки тысяч «mixed».

Вывод

На синтетике красиво, на реальном корпусе — шум; v1 как фильтр непригоден.

03

Аудит FastText mixed

Baseline
Подход

Эвристика is_real_mixed на строках с меткой mixed.

Результат

Только ~0% FastText-mixed — настоящий code-switch; ~0 seed-строк; ещё ~0 mixed внутри ru.

Вывод

Precision катастрофически низкая — нужны новые данные и модель.

04

FastText v2 (real seeds)

Baseline
Подход

Дообучение на synthetic + 0 verified mixed + kz/telegram.

Результат

Precision на real mixed 0% (0/0); старая v1 — 0%.

Вывод

Сильный скачок, но для 3-class LID и границы kz ↔ mixed всё ещё мало — идём в трансформер.

05

Lingua (token-level)

Baseline
Подход

Токенный ru/kz/mixed; кандидаты для ручной разметки.

Результат

~5–6k mixed-кандидатов — меньше шума, чем FastText; ~0 вручную вошли в gold.

Вывод

Полезен для добычи кандидатов, не как финальный LID без gold.

06

LLM-разметка LID (Qwen 2.5 3B)

Отклонён
Подход

Автоматическая разметка через Ollama / LLM в labeler.

Результат

Много false positive mixed (как у FastText).

Вывод

Слабая LLM не заменяет ручной gold; labeler оставили для ручной разметки и LLM-draft тона.

07

Gold LID + XLM-R

Production
Подход

0 ручных меток (mixed 0 · ru 0 · kz 0), labeling_service; fine-tune xlm-roberta-base.

Результат

v1 macro-F1 0% → v2 0%, acc 0%; ошибки в основном kz ↔ mixed, ru без ошибок на test n=461.

Вывод

xlm-r_v2.pt — production LID; фильтр корпуса → main_mixed.csv ~0k mixed (model-predicted).

08

Корпус и 2GIS

Baseline → данные
Подход

Telegram ~0k, Kaspi ~0k, 2GIS (Алматы, Астана, Шымкент, Атырау); main.csv ~0k.

Результат

Mixed для tone — отзывы 2GIS после LID-фильтра.

Вывод

Домен отзывов + соцсети; gold tone — mixed 2GIS, не весь Telegram.

09

Tone: gold → Mixed v1 + pretrained RU/KZ

Production Pretrained
Подход

Tone gold 0 (llm_composer + manual audit) + 0 synthetic (0%); 0 train; binary pos/neg.

Результат

Mixed Tone v1 — лучший чекпоинт (v2 хуже); RU/KZ — pretrained (RuReviews RuBERT, Kazakh Sentiment BERT), не на mixed-датасете.

Вывод

Mixed тон — своя модель; ru/kz — open-source routing; каскад LID → route → tone.

Итог: шала-казахский нельзя надёжно ловить правилами или FastText без gold. Рабочая схема — ручной gold → XLM-R LID v2 → маршрутизация тона (mixed — наша модель, ru/kz — pretrained). Ограничения: kz ↔ mixed, ~13% слабый code-switch в tone-gold, RU/KZ tone без fine-tune на наших данных.

01

04

Маршрутизация

Ограничения

Зрелый проект называет слабые места вслух.

Что мы знаем честно

  • ~13% gold mixed tone — слабый code-switch
  • FastText mixed в сыром Telegram: ~1,66% истинного mixed (эвристика is_real_mixed)
  • Gold LID 3k — research POC, не production scale
  • RU / KZ Tone — pretrained HuggingFace, не на нашем корпусе; 3 класса vs бинарный mixed
  • Tone gold: домен ресторанов 2GIS + llm_composer (3 312) + ручная разметка (217)

О проекте

KazNLP исследует шала-казахский в соцмедиа и отзывах Казахстана: отделить истинный code-switch от ложного «mixed» и классифицировать тон специализированной моделью.
