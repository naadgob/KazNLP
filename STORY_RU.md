## Главная мысль

Автоматический language ID в казахстанских соцсетях и отзывах помечает почти всё как «mixed». KazNLP измерил этот шум на **422 141** комментарии Telegram, собрал ручной эталон из **3 076** строк и на едином gold test (**n = 461**) показал, что XLM-RoBERTa v2 даёт **96,56%** macro-F1 там, где FastText останавливается на **70,92%**, а Lingua не годится для фильтрации корпуса без ручной проверки. Тональность, корпус **331 468** строк и demo строятся вокруг этого результата, не наоборот.

# I. Проблема в реальном мире

## Где это живёт

В Telegram-каналах, на Kaspi и в 2GIS люди пишут так, как говорят: русская фраза, потом казахская, иногда наоборот. Шала-казахский для миллионов пользователей — норма, не сбой клавиатуры. Для системы, которой нужно определить язык и тон, это отдельный класс задач.

Корпус проекта собран из реальных источников: **422 141** комментарий Telegram, **39 129** отзывов Kaspi, десятки тысяч отзывов 2GIS. Это не учебный датасет, а KZ digital как он есть.

## Симптом, причина, корень

**Симптом.** Отчёт по «доле mixed в соцмедиа» рисует картину массового двуязычия. Маркетолог строит дашборд. Исследователь кладёт тексты в корпус для sentiment. Продуктовая команда принимает решения по цифрам, которые при сборе уже испорчены.

**Причина.** FastText при сборе и эвристики «казахские буквы + кириллица» не отличают code-switch от казахского предложения с русским заёмным словом. В отчёте оценка ложных mixed — до **~98%** на уровне таких правил. В ноутбуке на конкретном срезе: из **27 628** FT-mixed эвристика `is_real_mixed()` пропустила **460** строк (~**1,66%**). После FT v2 precision mixed по той же эвристике — **2,32%** (1 542 из 66 462).

**Корень.** Казахстанский NLP годами смешивает два разных явления: монолингвальный казахский с заёмами из русского и настоящее переключение языков в одном сообщении. Пока граница не проведена, статистика по mixed, фильтры для тональности и любой «анализ двуязычия» опираются на шум.

### Два примера, без которых проект не объяснить

| Текст | Что это | Что делает наивный LID |
|-------|---------|------------------------|
| «Курьер молодец, уақытында әкелді» | **mixed** — две языковые фразы | Часто верно |
| «Качествосы жақсы, арзан» | **kz** — казахская грамматика, русское слово | Часто ошибочно **mixed** |

Вторая строка даёт ложные срабатывания в масштабе. Наивный LID видит кириллицу плюс «казахские» буквы и ставит mixed; заём в казахском выглядит как переключение.

Пока нет честного разделения ru / kz / mixed, нельзя ответить на простые вопросы: сколько в корпусе настоящего двуязычия, можно ли учить тональность на «смешанных» текстах, не подмешивая kz с заёмами. Если LID врёт, врёт и аналитика.

**Желаемое состояние:** фильтр трёх классов на реальных KZ-текстах, затем тональность только там, где язык действительно смешан.

**Цена бездействия:** **422 141** комментарий с меткой `language` от FastText при сборе (это **не** ground truth). То же на **39 129** отзывах Kaspi. Модели и отчёты на auto-labels учатся на шуме.

## Что решает KazNLP

Не «определение языка вообще», а узкая задача для KZ:

- Точный фильтр **ru / kz / mixed** с измеримым качеством на ручном gold.
- Честная оценка масштаба code-switching: сколько mixed реально, а не сколько написал FastText при сборе.
- Основа для тональности на смешанных отзывах (stretch goal, домен 2GIS mixed).
- Воспроизводимый pipeline от корпуса до demo API.

## Какую боль закрывает

| Боль | Без KazNLP | С KazNLP |
|------|------------|----------|
| «Сколько у нас mixed?» | Ответ из auto-labels, порядка **~98%** шум | Диагностика **1,66% / 2,32%** + gold test **96,56%** |
| Обучение на «смешанных» | В выборку попадает kz с заёмом | Filter-first: LID v2, потом tone |
| Сравнение подходов | Каждая модель на своём eval | Одна лестница, test **n = 461** |
| Воспроизводимость | «Открой ноутбук и верь» | `eval_tone_v1.py`, веса на диске, `run_demo.py` |
| Разметка low-resource | Ручная работа в CSV | `labeling_service`: очередь, экспорт gold |

Закрывается конкретный зазор: нельзя честно работать со смешанными KZ-текстами, пока LID врёт.

## До, после, путь

**До начала проекта**

- Telegram **422k**, колонка `language` от FastText при сборе, не эталон.
- FastText v1: F1 **84,96%** на **96k** синтетики. Задача казалась решённой.
- Реальный Telegram: почти все FT-«mixed» — ложь.
- Lingua v2: на seeds **95,97%**, на gold test recall mixed **98,76%**, precision **76,81%** — для фильтра корпуса опасно.
- Нет gold для шала-казахского в масштабе проекта.
- Тональность на mixed: неясно, на каких текстах учить.

**Путь (как пришли к решению)**

Измерили шум → отвергли короткие пути (синтетика, FT v2 на корпусе, Lingua) → собрали gold **3 076** → обучили XLM-R на сыром `text` → доказали на едином test §10 → применили к корпусу → добавили tone и продукт.

```
ДО:   auto-LID → «всё mixed» → аналитика и модели на шуме
        ↓
ПУТЬ: диагностика → gold → XLM-R v2 → ladder §10
        ↓
ПОСЛЕ: filter-first cascade, 331k с метками LID v2
```

**После проекта**

- Gold LID **3 076**; val/test только gold; synthetic только в train.
- XLM-R LID v2: **96,56%** macro-F1; mixed P/R ~**95%** на test **n = 461**.
- Baseline ladder: FT v2 **70,92%** → HeLI raw **69,73%** → Lingua v2 **88,63%** → XLM-R **96,56%**.
- `main.csv` **331 468** строк; **16 364** model-predicted mixed (не human-audited).
- Tone v1: **97,33%** на test **n = 525** (см. оговорку про LLM-labels ниже).
- Cascade TEXT → LID → route → tone; demo и labeler работают локально.

## Кому без решения хуже

Исследователю NLP нельзя опубликовать честную цифру по code-switching в KZ corpora. Аналитику соцмедиа дашборды по языкам врут на порядок. Команда, которая учит sentiment на mixed, получает kz-with-loanword вместо code-switch. На защите без измеренной проблемы проект выглядит как «ещё один BERT на 97%». KazNLP закрывает исследовательский и методологический зазор; остальные могут использовать фильтр и открытый pipeline.

## Почему сейчас

XLM-R видит контекст целой фразы; FastText и правила на **422k** реальных текстов уперлись в потолок. KZ digital вырос (Telegram, маркетплейсы, геосервисы). Без gold сильную модель жюри не докажешь: **3 076** ручных меток и один test важнее ещё **100k** auto-labels.

# II. Хронология исследования (`main.ipynb`)

Хронология ноутбука — логика проекта. Ниже — главы в порядке выполнения.

## Глава 1. Ложная уверенность синтетики (cells 0–21)

Старт: Hugging Face (KazSAnDRA **180 064**, clapAI ru **164 148**), **480 000** синтетических ru/kz/mixed с словарями `popular_ru_kaz` и `essential_ru_kaz`, обучение FastText v1.

На hold-out **96 000** строк F1 **84,96%**. На бумаге LID почти решён. Дальше — реальный Telegram.

## Глава 2. Реальный корпус ломает картинку (cells 22–44)

Сбор Telethon по KZ-каналам; FastText v1 ставит `language` при сборе. Датасет рос: **146 206** → **241 576** → **422 141** на диске.

Визуально почти всё «mixed» выглядит как русский или казахский с заёмом. Ключевой вопрос: сколько из auto-mixed — настоящий code-switch?

## Глава 3. Диагностика (cells 45–64)

Сначала измерили шум, не трансформер.

`is_real_mixed()` на FT-mixed: **460** из **27 628** (~**1,66%**). **411** скрытых mixed в `language=ru`. Пул **868** seeds (не gold; пропускает shala-Kazakh без ә/ң).

FT v2 на синтетике + seeds: recall **95,51%** (829/868) на seeds. На корпусе после relabel **66 462** mixed, precision **2,32%** (1 542/66 462).

Короткие пути не работают. Синтетика даёт красивый F1 и не переносится на реальные тексты.

## Глава 4. Правила не спасают (cells 65–113)

Lingua v1 на 868 seeds: **71,66%**. Lingua v2: **95,97%** на seeds; полный проход **411k** Telegram → **6 517** кандидатов, ручная проверка всё равно нужна.

На gold test (**n = 461**, §10): Lingua v2 macro-F1 **88,63%**, recall mixed **98,76%**, precision **76,81%**. Высокий recall при низком precision: фильтр корпуса без проверки снова зальёт ложными mixed.

Параллельно Kaspi **39 129**. YouTube (~4,7k) не доводили; хватило Telegram + Kaspi + 2GIS.

Нужны ручной эталон и модель, которая видит фразу целиком.

## Глава 5. Gold (cells 114–153)

Kaspi + Telegram, `cheap_clean`: **388 748** → **254 601** строка.

Gold батчами: ручная разметка, heuristic-batch (только mixed), очередь Lingua. После dedup **3 076**: ru **1 000**, kz **999**, mixed **1 077** → `gold_v1.csv`.

Протокол обучения:

- stratified split 80/10/10, `random_state=42`: train **2 691**, val **462**, test **461**;
- **538** synthetic hard-паттернов только в train;
- вес gold : synthetic = **3:1**;
- XLM-R на сыром **`text`**, не `text_norm` (~219/1077 mixed теряют сигнал после нормализации);
- LLM в gold LID не использовали (высокий false positive на mixed).

## Глава 6. Доказательство: XLM-R LID v2 (cells 154–176, §10)

Обучение: Kaggle 2× T4, AdamW lr 2e-5, XLM-RoBERTa-base.

| Версия | macro-F1 (gold test n = 461) |
|--------|----------------------------:|
| v1 | 95,92% |
| **v2** | **96,56%** |

Confusion matrix v2 (rows = true, cols = pred):

| | pred ru | pred kz | pred mixed |
|--|--------:|--------:|-----------:|
| true ru | 150 | 0 | 0 |
| true kz | 0 | 142 | 8 |
| true mixed | 2 | 6 | 153 |

**14 из 16** ошибок: путаница kz ↔ mixed.

§10 (cells 267–271), один `test.csv`:

| Модель | macro-F1 |
|--------|--------:|
| FastText v1 | 63,24% |
| FastText v2 | 70,92% |
| HeLI raw (heliport) | 69,73% |
| HeLI+neutral | 68,26% |
| Lingua v1 | 84,96% |
| Lingua v2 | 88,63% |
| **XLM-R v2** | **96,56%** |

Только XLM-R v2 даёт balanced mixed P/R ~**95%** на hold-out. Это главный научный результат capstone. HeLI (Tommi Jauhiainen / heliport, опционально strip заимствований) стоит рядом с FastText как не-нейронный baseline; нейтрализация на этом сплите raw HeLI не обогнала.

## Глава 7. Применение к корпусу (cells 177–237)

XLM-R v2 перескорил Kaspi+Telegram → `kaspi-telegram_dataset_v2.csv` (**254 652** на диске).

2GIS: negative **29 479**, positive **35 928** после чистки. Финальная сборка только в **cell 237** (промежуточные ячейки 203/221 содержат устаревшие счётчики):

**`main.csv` — 331 468** (ru **281 409** · kz **33 695** · mixed **16 364**).

`main_mixed.csv` — предсказания модели, не ручной аудит. Corpus audit 100 random mixed из Action Plan не выполняли. Формулировка «16 тысяч подтверждённых mixed» некорректна; корректно: «16 364 кандидата по LID v2, precision на корпусе не измерялась».

## Глава 8. Stretch goal: тональность (cells 238–264)

Отдельная задача, домен 2GIS. Gold tone: **3 529** mixed (pos **1 771**, neg **1 758**); `llm_composer` **3 312**, manual **217**. + **882** synthetic в train → **4 411** в merge; test **525** gold-only.

Tone v1: accuracy **97,33%**, CM `[[257, 6], [8, 254]]` — совпадает с `metrics_tone_test.json` и `scripts/eval_tone_v1.py`. v2 хуже (**96,19%**), в production выбран v1.

**Оговорка:** ~**94%** tone gold создано с LLM-draft; на test **488 из 525** строк тоже `llm_composer`. Метрика **97,33%** отражает согласованность с LLM-разметкой на hold-out, а не независимую человеческую валидацию. Manual subset (**217** строк) отдельно не оценён.

## Глава 9. Продукт

Архитектура filter-first (`inference/pipeline.py`):

```
сырой text → XLM-R LID v2 (ru | kz | mixed)
                    ↓
              auto-route
       ┌──────────┼──────────┐
       ru         kz        mixed
       ↓          ↓           ↓
   RuBERT      Kazakh BERT   Tone v1 (XLM-R)
  (pretrained) (pretrained)  (fine-tuned)
```

Для ru/kz тональность идёт через pretrained-модели; для mixed — fine-tuned XLM-R tone v1.

Воспроизведение:

```bash
python scripts/setup_demo_models.py
python run_demo.py              # http://127.0.0.1:8000/
python run_labeler.py           # разметка gold
python scripts/eval_tone_v1.py  # tone-метрики
```

Первый запрос к API может вернуть 503 на 30–60 с (загрузка четырёх моделей, ~8,56 GB).

# III. Ограничения и итог

## Честные границы

- Один разметчик gold LID; согласие между двумя разметчиками не измерял.
- `main.ipynb` (272 ячейки) не воспроизводится через Run All; ячейки Telethon **31, 35, 41, 88** нельзя перезапускать.
- Defense path: артефакты на диске, cells **45, 173, 237, 268**, `eval_tone_v1.py`, `run_demo.py`.
- Tone и LID оценивались на разных доменах (Telegram/Kaspi vs 2GIS).
- **16 364** mixed в `main.csv` не проходили ручную проверку на корпусе.

Это границы student-scale исследования; подробнее — `Final_Report_RU.md` §3.5, §4.2.

## Итог

Проект начался с вопроса «почему все пишут mixed». Ответ оказался в измерении: **~1,66%** на диагностическом срезе, **~2,32%** precision на корпусе после FT v2, gold **3 076**, XLM-R v2 **96,56%** на test **n = 461**. Tone **97,33%** на mixed 2GIS — stretch goal, вторичен относительно LID. Demo и labeler показывают, что pipeline вышел за рамки ноутбука.

**Краткая формулировка (30 секунд):**

> В KZ соцмедиа auto-LID помечает почти всё как mixed. KazNLP измерил шум на 422k Telegram, собрал gold 3 076, довёл XLM-R v2 до 96,56% macro-F1 на честном test и собрал cascade LID → тональность с локальным demo.

**До:** статистика и модели на шуме. **После:** filter-first pipeline с доказанным LID и явно названными ограничениями.
